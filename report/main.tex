\abstract{This paper presents a deep learning approach to classifying the composer of a
  classical work directly from symbolic MIDI data. Using the \textit{MIDI Classic Music}
  dataset from Kaggle \parencite{Blanderbuss2025}, filtered to four composers
  (Bach, Beethoven, Chopin, and Mozart), we extract note- and performance-level
  features and compare two PyTorch architectures: a long short-term memory (LSTM)
  network operating on note sequences and a convolutional neural network (CNN)
  operating on piano-roll-style matrix representations. Because the filtered corpus
  is imbalanced, we use class-weighted cross-entropy loss and data augmentation for
  minority classes. Models are evaluated with accuracy, precision, recall, and
  confusion matrices on held-out test data.}

\keywords{composer classification, MIDI, deep learning, LSTM, CNN, music information retrieval}

\section{Introduction}
Automatic composer identification is a long-standing music information retrieval
problem: given a piece of music, predict which composer wrote it. Symbolic
representations such as MIDI expose pitch, duration, and timing information
directly, without the confounds of audio timbre or recording conditions, making
them a natural substrate for sequence and pattern-based deep learning models.

\section{Related Work}
\textcite{Hochreiter1997} introduced the long short-term memory architecture,
which remains a standard approach for modeling long-range dependencies in
sequential data such as musical note sequences. \textcite{LeCun2015} describe
convolutional architectures that are well suited to grid-structured
representations such as piano rolls. \textcite{Kingma2015} introduced the Adam
optimizer used to train both models in this study.

\section{Method}

\subsection{Dataset}
The dataset used for this project is the \textit{MIDI Classic Music} corpus from
Kaggle \parencite{Blanderbuss2025}. The archive contains MIDI files for many
classical composers; for this study we filtered it to the four required
composers: Bach, Beethoven, Chopin, and Mozart. The helper script
\texttt{src/prepare\_dataset.py} downloads the archive (when
\texttt{{\raise.17ex\hbox{$\scriptstyle\sim$}}/.kaggle/kaggle.json} is
configured), extracts the four composer folders, and creates reproducible
training, validation, and test splits using a fixed random seed. Class balance
across the filtered corpus is shown in Figure~\ref{fig:class-distribution}.

\begin{figure}[!htp]
  \centering
  \caption{Class Distribution Across the Four Composers}
  \includegraphics[width=0.8\linewidth]{../figures/class_distribution.png}
  \label{fig:class-distribution}
\end{figure}

The filtered dataset is imbalanced: Bach accounts for roughly 63\% of training
files (717 of 1,141), whereas Chopin contributes only 8\% (95 of 1,141). To
prevent the model from simply predicting the majority class, we use
class-weighted cross-entropy loss during training and apply pitch-shift and
time-stretch data augmentation to the minority classes. We also report
per-class precision and recall alongside overall accuracy.

Three findings shaped modeling decisions. First, note density differs
substantially across composers---Bach's counterpoint averages 9.5 notes/s while
Beethoven reaches 14.0 notes/s---making density a stronger discriminator than
raw note count, which correlates strongly with file duration. Second, pitch
range separates eras: Bach's Baroque writing occupies a narrower tessitura
($M = 42.8$ semitones) than the Romantic-era Beethoven and Chopin
($M \approx 62$ semitones), consistent with idiomatic keyboard-writing
conventions. Third, tempo and note velocity often reflect MIDI-encoding
defaults rather than genuine performance data, so neither is a reliable
standalone feature.

A principal component analysis of the engineered statistical features showed
only modest separation between composer classes (Figure~\ref{fig:pca}),
indicating that hand-crafted summary statistics alone are insufficient for
reliable classification. This motivated the sequence-based (LSTM) and
image-based (CNN, piano-roll) representations described below, which preserve
temporal and structural information the aggregate statistics discard.

Quantitatively, the first principal component explained 40.1\% of variance and
the second explained 14.7\% (54.8\% cumulative), which is not enough to produce
clear class clusters in two dimensions. Correlation analysis also indicated
that raw note count and duration are strongly coupled, so we prioritize derived
features such as note density and temporal representations that preserve
rhythmic structure rather than relying only on aggregate counts.

\subsection{Model Architectures}

\subsubsection{LSTM Model}
The LSTM model consumes sequences of note events (pitch, duration, velocity) and
predicts a composer label from the final hidden state.

\subsubsection{CNN Model}
The CNN model consumes fixed-size piano-roll matrices and predicts a composer
label using stacked convolutional and pooling layers.

\subsection{Training}
Both models are trained with the Adam optimizer \parencite{Kingma2015}. Because
the filtered dataset is imbalanced, we use class-weighted cross-entropy loss
with inverse-frequency weights computed on the training split. We additionally
augment minority-class training files with random pitch shifts ($\pm 1$ to
$\pm 3$ semitones) and small time-stretch factors to improve generalization.
Training stops when validation loss fails to improve for a fixed number of
epochs, and the best validation checkpoint is retained for evaluation on the
held-out test set.
